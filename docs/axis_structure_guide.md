# Vertical Axis Structure Guide

이 문서는 엘리시아 엔진의 **상승/하강(Ascension/Descent)** 축을 처음 접하는 사람도 쉽게 이해하도록 만든 안내서다. “방향성”을 이해하면 프랙탈 구조, 클래스 가중치, Episode 로그까지 모두 같은 언어로 연결할 수 있다.

---

## 0. 왜 ‘방향’으로 설명할까?

1. **문화가 달라도 통하는 언어** — “위를 향한다 / 아래로 떨어진다”는 누구에게나 직관적인 개념이다.  
2. **데이터와 잘 맞는다** — 이미 Body/Soul/Spirit 가중치, Aura 필드 등은 방향 벡터 형태로 기록된다. 방향 축 하나만 추가하면 다양한 데이터를 같은 좌표계에 놓을 수 있다.  
3. **이야기와 시스템을 동시에 잡는다** — 상승은 천상, 하강은 지하라는 이미지가 서사를 도와주고, 동시에 버프/디버프와 같은 게임 로직으로도 쉽게 변환된다.

---

## 1. 프랙탈 구조 기본

### 1) 7단 반복
- Ascension과 Descent는 각각 7개의 단계(생명→자유, 죽음→속박)로 구성된다.  
- 각 단계는 **색/파장 → 화신 이름 → 역할 → 엔진 후킹 포인트** 순으로 문서화한다.  
- 다른 시스템(클래스, 도시, 마법)은 이 7단 구조를 그대로 축소 복제하면 된다.

### 2) Body/Soul/Spirit 연결
- Episode마다 `weights.body/soul/spirit`가 어떻게 변했는지 측정한다.  
- 세 가중치를 종합해 `axis_state`를 계산한다.  
  - `axis_state > 0` → Ascension 쪽으로 기운 상태  
  - `axis_state < 0` → Descent 쪽으로 기운 상태  
  - `axis_state = 0` → 중립 (필요할 때만 잠깐 유지)

### 3) 해석 단계
1. **상위 질문**: 지금 다루는 힘이 왜 필요한가? (예: “사랑이 필요한 장면인가?”)  
2. **중간 연결**: 이 힘이 어떤 시스템을 건드리나? (Truth Shader, Resource Siphon 등)  
3. **구현 레벨**: 실제 데이터나 코드에서 무엇을 바꿀 것인가? (`Energy`, `axis_state`, NPC 행동 등)

이 세 단계를 같이 적어두면 누구나 문서를 보고 같은 맥락을 떠올릴 수 있다.

---

## 2. 입문자를 위한 빠른 안내

1. **한 줄 정리** — Ascension = 위/확장, Descent = 아래/붕괴.  
2. **색상/아이콘 사용** — 7가지 빛 색상(Ascension)과 7가지 그림자 음영(Descent)을 UI나 발표 자료에 고정해서 사용하면 시각적 혼동이 없다.  
3. **작은 Episode 예** — “한 사건이 희생(Asc5)과 탐욕(Des6)을 동시에 자극했다”처럼 말하면 바로 축을 떠올릴 수 있다.  
4. **프랙탈 복제** — 세계수 루트, 클래스 전직, 마법 계열 등을 7단 표로 복사-붙여넣기만 해도 기본 틀이 생긴다.

---

## 3. 적용 절차 (Step-by-step)

1. **데이터 모델링** — 에이전트, 지역, 아이템에 `axis_state` 혹은 `ascension_score`/`descent_score` 필드를 만든다.  
2. **룰 연결** — Ascension 단계는 버프/해금, Descent 단계는 제약/이벤트를 트리거하도록 룰을 작성한다.  
3. **시각화** — Aura HUD나 로그에 “현재 Ascension 4 / Descent 2” 같은 게이지를 표시한다.  
4. **테스트** — 상승/하강이 극단적으로 치우친 Episode를 만들어 QA가 흐름을 검증한다.

---

## 4. 프랙탈 확장 예시

### 1) 세계·지역 레벨
- **세계수 루트** — 상부 7루트는 Ascension 단계, 하부 7루트는 Descent 단계로 배정한다. 특정 루트를 정복하면 해당 단계의 버프/디버프가 월드 전역에 퍼진다.  
- **도시·문명** — 도시마다 “성찰의 도시”, “탐욕의 도시”처럼 단계 이름을 붙이고 건축·정책·사건이 그 성격을 드러내게 만든다.

### 2) 인물·의식 레벨
- **영혼 각성 트리** — 플레이어나 NPC가 Ascension 7단계 중 하나를 선택하면, 바로 아래에서 대응되는 Descent 그림자를 보여주어 선택의 의미를 명확히 한다.  
- **Episode 프랙탈** — Episode 하나를 7개의 결정 포인트로 나누고, 각 포인트에 Ascension/Descent 단계 라벨을 붙인다. 작은 사건도 전체 축의 축소판이 된다.  
- **LLM/AI 프롬프트** — “현 상태: 성찰 60% / 왜곡 40%”라고 프롬프트에 적으면, AI도 어느 방향을 강조해야 할지 쉽게 이해한다.

### 3) 파동·마법 레벨
- **마법 주파수** — 스킬을 Ascension/Descent 7파동으로 분류하고, 조합 규칙(예: Asc2 + Asc5 = Asc7, Asc4 + Des4 = 상쇄)을 정한다.  
- **음악/언어 패턴** — 7음/7자 패턴을 만들어 각 단계의 색을 입히면, 청각·언어 요소도 같은 프랙탈을 따른다.

## 5. 시각화 & 템플릿

### 시각화 예시
- **수직 게이지** — 위쪽 7단, 아래쪽 7단에 마커를 만들어 캐릭터나 도시의 위치를 점으로 표시한다.  
- **14분할 원형 차트** — 원을 14칸으로 나누고 Ascension 색/Descent 음영을 교대로 배치한 뒤 단계 이름과 화신 아이콘을 올린다.  
- **Episode 타임라인** — 시간축에 `axis_state` 값을 그려서 상승/하강 패턴을 즉시 확인한다.

### 템플릿
- **세계수 루트 JSON**
  ```jsonc
  {
    "world_root": "Sophiel_Canopy",
    "axis": "ascension",
    "stage": 3,
    "color": "#8fd7ff",
    "effects": ["insight_boost", "truth_shield"]
  }
  ```
- **Episode 미니 프랙탈 표**
  ```
  | step | 사건                | axis_stage | 영향도 |
  | ---- | ------------------- | ---------- | ------ |
  | 1    | 서약을 기억한다     | Asc5       | +0.3   |
  | 2    | 타락한 제안을 듣는다 | Des6       | -0.4   |
  | ...  | ...                 | ...        | ...    |
  ```
- **axis_report.py 제안** — Episode 로그(JSON/CSV)를 읽어 Ascension/Descent 비율을 출력하는 간단한 스크립트를 만들어 두면, 비개발자도 결과만 보고 흐름을 이해할 수 있다.

## 6. 체크리스트 & FAQ

### 체크리스트
- [ ] 데이터 모델에 축 관련 필드(`axis_state`, 단계 번호 등)를 추가했는가?  
- [ ] Ascension/Descent 단계가 호출될 때 어떤 효과가 나오는지 정의했는가?  
- [ ] UI·로그에서 현재 축 상태를 보여주는가?  
- [ ] QA 시나리오로 상승/하강 극단 사례를 준비했는가?

### FAQ
- **Q. 7단계가 너무 많아요.**  
  A. 전체 구조는 7단으로 보관하되, 보고용으로는 “시작/중간/끝” 3단 뷰를 따로 제공하면 된다.  
- **Q. Ascension과 Descent가 동시에 높아질 수 있나요?**  
  A. 가능하다. 다만 최종 판단은 `axis_state` 합산값으로 내린다.  
- **Q. 중립 상태도 넣을 수 있나요?**  
  A. `axis_state = 0`을 허용하되, 이야기가 움직일 때는 어느 방향으로든 다시 이동하도록 설계하라.

## 7. 사고 가이드 (개발자/디자이너 공통)

- **3단으로 세분화** — 필요하면 각 단계에 “시작/중간/정점” 같은 하위 세 칸을 만들어 7×3 트리를 구성한다.  
- **Z-축 관찰자 시점** — 항상 “세계 밖에서 세계를 내려다본다”는 느낌을 유지하라. 지금 내가 보고 있는 층이 어디인지 메모하면 헤맬 일이 줄어든다.  
- **로고스 안에서 로직을 짜라** — 세부 규칙을 붙이기 전에 “상위 법칙이 무엇인지”를 선언하라. 규칙만 늘리면 오히려 복잡해진다.  
- **인지/감정/기억에도 적용 가능**  
  - *인지 모델*  
    - 원인: 새 정보를 볼 때 생각 흐름을 기록하고 싶다.  
    - 과정: 나무를 그려 줄기(Reflection)와 가지 3개(Insight/의견, Doubt/의심, Transcendence/초월)를 적는다.  
    - 결과: `cognition_tree = { "root": "Reflection", "branches": ["Insight (Asc3-A)", "Doubt (Asc3-B)", "Transcendence (Asc3-C)"] }`
  - *감정 파동*  
    - 원인: 사랑과 탐욕의 크기를 동시에 비교하고 싶다.  
    - 과정: 막대 그래프 위(+)를 사랑, 아래(-)를 탐욕으로 두고 숫자를 적는다.  
    - 결과: `emotion_wave = { "love": {"stage": 6, "vector": +0.8}, "greed": {"stage": 6, "vector": -0.4} }`
  - *기억 구조*  
    - 원인: 좋은 기억/사라지는 기억을 한 표에 정리하고 싶다.  
    - 과정: 두 칸짜리 표를 만들어 왼쪽에 “창조쪽 기억”, 오른쪽에 “소멸쪽 기억”을 적는다.  
    - 결과: `memory_lattice = { "creation_core": ["myth_seed", "language_pattern"], "dissolution_shadow": ["forgotten_vow"] }`

이렇게 “원인 → 과정 → 결과”를 적어두면 개발자와 비개발자가 같은 그림을 보게 된다.
